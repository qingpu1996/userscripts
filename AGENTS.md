# 开发约束

本文件适用于整个仓库。这里只记录长期有效的开发协作、验收、版本控制收尾和开发文档归档规则。

## 1. 主会话、开发 SubAgent 与验收 SubAgent

### 1.1 主会话职责

- 主会话只负责需求澄清、任务拆分与派发、状态跟踪、验收移交，以及验收通过后的版本控制、按需 PR/CI 与合并收尾。
- 主会话不得直接实现需求或修改项目文件；默认在主会话内部派发一次性开发 SubAgent 和验收 SubAgent，并保持两个角色严格隔离。
- 主会话负责规划分支和 worktree、核对开发与验收回告、确认相关 SubAgent 完成回告后已经结束/关闭，并向用户统一回告。

### 1.2 一次性开发 SubAgent

- 每个可拆分的开发事项默认新建一个边界清晰的一次性开发 SubAgent；不得复用主会话直接开发，也不得让同一个 SubAgent 同时承担开发与验收。
- 大型事项可以拆给多个开发 SubAgent，但每个 SubAgent 都必须有独立且不冲突的范围。如需进一步拆分子任务，负责派发的 Agent 必须写明允许修改的文件或目录、禁止事项和输出要求，并对其负责范围内的完整实现、验证、下级 SubAgent 收尾和最终回告负责。
- 下发给开发 SubAgent 的任务必须至少包含：背景、明确终止目标、需求与范围、允许修改的文件或目录、禁止事项、验收标准、验证要求、完成后的回告格式。
- 开发完成后的回告必须至少包含：变更摘要、修改文件、验证命令与结果、未覆盖风险、待确认问题，以及用于移交的当前分支、worktree 绝对路径、HEAD/commit 和是否存在未提交改动等工作区状态信息。
- 开发 SubAgent 完成回告后必须立即结束/关闭，不作为常驻任务保留；主会话必须确认其已结束，再继续验收移交。
- 开发 SubAgent 不得执行 stage、commit、push、追加 PR comment、rebase、squash 或 merge；这些操作只允许在验收通过后由主会话统一处理。

### 1.3 验收移交

- 开发 SubAgent 完成回告并结束后，主会话必须新建另一个未参与开发的只读验收 SubAgent；不得复用开发 SubAgent 自验，也不得由主会话自行验收代替。
- 验收 prompt 的第一行必须是以下原句，前面不得添加其他文字：

  ```text
  请作为验证线程，不要继续开发，只审查这次执行结果。请优先输出问题，再说明剩余风险和是否可以交付
  ```

- 验收 SubAgent 的 prompt 必须完整附带：原开发任务的背景、终止目标、需求与范围、允许修改的文件或目录、禁止事项、验收标准、验证要求、开发回告、当前分支、worktree 绝对路径、HEAD/commit、是否存在未提交改动等工作区状态信息，以及 QA 证据位置。

### 1.4 全新只读验收 SubAgent

- 验收 SubAgent 必须是全新的、未参与开发的 SubAgent，并严格只读；只审查执行结果并按“问题、剩余风险、是否可以交付”的顺序回告，不得修复、编辑、stage、commit、push、追加 PR comment、rebase、squash 或 merge。
- 验收 SubAgent 完成回告后必须立即结束/关闭，不作为常驻任务保留；主会话必须确认其已结束，并记录验收结论。
- 验收发现问题时，主会话必须新建一个边界清晰的一次性修复 SubAgent 处理，不得复用已完成回告并结束的原开发 SubAgent。修复 SubAgent 完成修复、验证和回告后必须立即结束/关闭；主会话确认其已结束后，必须再新建一个全新的只读验收 SubAgent 复核，不得复用上一验收 SubAgent，也不得由主会话代替。

### 1.5 验收通过后的统一收尾

- 只有验收 SubAgent 明确确认可以交付且主会话确认其已结束后，才由主会话统一执行 stage、commit、push、按需创建或更新 PR、rebase、squash 与 merge 等收尾操作。
- 独立 Codex 验收是唯一强制审查门禁；push 前仍需用户明确确认。PR 仅在需要远程留痕、协作或 CI 时按需使用；任务要求的 CI 必须通过后才可合并。
- 如果收尾过程中出现需要修改项目文件的冲突或新问题，主会话必须停止收尾并派发一次性修复 SubAgent；修复后再新建全新的只读验收 SubAgent 复核。

### 1.6 用户可见新会话的例外

- 默认不得为每个开发或验收事项创建用户可见的新 Codex 会话，以免项目列表被一次性任务占满。
- 只有用户明确要求，或任务确实需要长期可见、由用户接手、跨时段持续协作时，才可创建用户可见的新 Codex 会话。
- 例外使用用户可见新会话时，仍须遵守开发与验收角色隔离、全新只读验收唯一强制门禁、完成回告字段、push 前用户确认，以及主会话统一收尾等全部规则；用户可见性本身不能替代验收。

## 2. 开发文档归档规则

- 根目录 `AGENTS.md` 只放长期有效的项目级开发约束、协作流程和索引。
- 需求、验收、阶段计划、视觉规格、参考代码和调试说明应迁到专题文档。
- 项目总览、历史设计和其他长期开发文档统一放在 `docs/`。
- 不要直接删除仍约束当前实现的需求；先迁移，再在 `AGENTS.md` 保留链接。
